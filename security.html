Security Documentation

Cryptographic Specifications

Algorithm AES-256-GCM
Key Derivation Function PBKDF2-HMAC-SHA256 (100,000 iterations)
Initialization Vector (IV) 96-bit random
Authentication Tag 128-bit
Key Size 256-bit

Double Encryption Architecture

Gob Goblet uses two layers of encryption for maximum security. The outer layer (client-side PBKDF2, completely offline) hides all metadata, including the original filename and file type. The inner layer (server-side PBKDF2 utilizing our secret pepper) protects the actual file bytes. An attacker who intercepts a .gob file cannot determine the original filename, file type, or any metadata without the password.

The Magic Header

Every .gob file begins with the %gob magic bytes, a version number, and a human-readable notice. This makes the files self-describing and identifiable by operating systems, while keeping all sensitive data completely encrypted.

Server Pepper

Our Cloudflare Worker holds a secret "pepper" that is mixed into the inner key derivation process. Even if an attacker obtains a .gob file and knows the password, they cannot derive the inner decryption key to access the file content without this server secret. This protects against offline brute-force attacks.

Threat Model

  • Mitigated: Network eavesdropping, offline brute-force dictionary attacks, metadata exposure (filenames/sizes), and file tampering (prevented via GCM authentication tags).
  • Not Mitigated: An adversary who obtains both the encrypted file AND compromises the Cloudflare Worker to extract the server secret, combined with a weak user password.

Trust Model

The user must trust that the open-source frontend code is served unaltered, Cloudflare hosts the worker securely without exposing environment variables, and the secret pepper is never committed to source control.
